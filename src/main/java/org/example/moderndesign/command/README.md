## Command 패턴

- 요청을 객체로 캡슐화하여, 실행하려는 작업을 나중에 실행하거나, 요청을 큐에 저장하거나, 실행 취소(undo) 기능 등을 가능하게 해주는 디자인 패턴

### Strategy vs Command 비교

- 목적 :
    - **Strategy**: 알고리즘을 캡슐화하여 런타임에 교체 가능하도록 하는 것
    - **Command**: 요청을 객체로 캡슐화하여 실행, 큐잉, 실행 취소 등을 가능하게 하는 것
- 초점 :
    - **Strategy**: 알고리즘의 교체와 유연성에 초점
    - **Command**: 요청의 실행과 관리에 초점
- 예시 :
    - **Strategy**: 결제 수단에 따라 결제 처리 로직을 다르게 구현 (신용카드, 페이팔 등)
    - **Command**: 결제 요청을 객체로 만들어 나중에 실행하거나, 실행 취소할 수 있도록 구현

### Strategy 패턴을 람다로 구현했을 경우의 단점

- 재사용성 낮음 : 람다는 보통 익명으로 정의되기 때문에, 동일한 명령을 여러 곳에서 재사용하려면 다시 작성해야 한다.
- 테스트의 어려움 : 명령 로직이 람다 안에 직접 들어있기 때문에, 단위 테스트 대상이 되기 어렵다. (Mocking 등도 어려움)
